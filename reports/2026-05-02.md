## Top 3 du jour

### 1. Attaque supply chain PyPI : PyTorch Lightning compromis, worm GitHub, impersonation Claude Code (9.5/10)

Le 30 avril 2026, les versions 2.6.2 et 2.6.3 du package `lightning` (PyTorch Lightning) ont été publiées sur PyPI avec une charge malveillante. Le payload — un script JavaScript obfusqué de 11 Mo exécuté via le runtime Bun — réalise un vol massif de credentials : tokens GitHub, clés d'API, secrets système. Les tokens volés sont validés contre l'API GitHub puis utilisés pour injecter une charge vermiforme dans jusqu'à 50 branches de chaque dépôt accessible. Détail frappant : chaque commit malveillant est signé en usurpant l'identité d'**Anthropic Claude Code**. La campagne est reliée à l'incident Mini Shai-Hulud (packages npm SAP). PyPI a mis le projet en quarantaine 18 minutes après publication, détection assurée par le scanner AI de Socket.

**Pourquoi c'est important :** Les obligations supply chain de NIS2 et DORA imposent désormais de cartographier et surveiller les dépendances tierces, y compris les packages open source utilisés dans les pipelines MLOps. L'angle d'usurpation de Claude Code est un signal fort : les attaquants ciblent spécifiquement l'écosystème IA/LLM, qui concentre des credentials à très haute valeur (clés OpenAI, Anthropic, AWS). Pour un alternant en cyber governance, c'est le parfait exemple concret à invoquer lors d'audits de chaîne d'approvisionnement logicielle.

### 2. CVE-2026-42208 (CVSS 9.3) : SQL injection pré-auth sur LiteLLM exploitée en 36 h (9/10)

LiteLLM, gateway open source pour LLM avec plus de 22 000 étoiles GitHub, est affecté par une injection SQL pré-authentifiée critique (CVE-2026-42208, CVSS 9.3) dans son chemin d'authentification proxy. La valeur du header `Authorization` était concaténée directement dans un `SELECT` sur la table `LiteLLM_VerificationToken` sans paramétrage. Un attaquant non authentifié peut lire et modifier la base, notamment les tables `litellm_credentials.credential_values` et `litellm_config` qui stockent les clés des providers LLM upstream (OpenAI, Anthropic, etc.). L'exploitation active a démarré 36 heures après la divulgation publique du 24 avril 2026. La version 1.83.7-stable corrige le problème.

**Pourquoi c'est important :** Les entreprises qui déploient LiteLLM comme proxy centralisé pour leurs équipes IA voient potentiellement toutes leurs clés API LLM compromises d'un seul coup — violation RGPD si des données personnelles transitent, et incident DORA si l'entité est dans le secteur financier. Le délai d'exploitation de 36 h illustre la nécessité d'un processus de patch management accéléré pour les composants IA exposés sur internet, un sujet encore absent de beaucoup de politiques de sécurité.

### 3. ANSSI publie le ReCyF : le référentiel de conformité NIS2 français est disponible (9/10)

Depuis le 17 mars 2026, l'ANSSI met à disposition le Référentiel Cyber France (ReCyF), qui décline les mesures recommandées pour atteindre les 20 objectifs de sécurité des entités essentielles et 15 objectifs des entités importantes fixés par NIS2. Ce référentiel, non-obligatoire par défaut, constitue néanmoins le standard de référence lors des contrôles ANSSI : une entité qui l'applique peut s'en prévaloir. Parallèlement, le vote en hémicycle de la loi Résilience (transposition française de NIS2, directive REC et DORA) est attendu pour juillet 2026. Le 18 mars 2026, le CEPD a adopté l'avis conjoint 4/2026 recommandant un point d'entrée unique pour les notifications d'incidents, afin d'harmoniser NIS2, DORA et RGPD.

**Pourquoi c'est important :** Pour les quelque 15 000 entités françaises qui entrent dans le périmètre NIS2 (contre 500 sous NIS1), le ReCyF devient la feuille de route opérationnelle immédiate. L'harmonisation des notifications (guichet unique) simplifie les obligations mais exige une révision des procédures de gestion des incidents existantes. En tant qu'alternant en cyber governance, maîtriser le ReCyF dès maintenant est un avantage concurrentiel direct dans les missions d'audit et de conformité.

## Autres news notables

- **CVE-2026-3854 — GitHub RCE via git push** (8/10) — Injection de commande dans le pipeline `git push` de GitHub Enterprise Server, CVSS 8.7 ; un simple `git push` avec des push options craftées permet l'exécution de code arbitraire côté serveur. Patché en urgence sur github.com le 4 mars 2026 en moins de 2 heures après signalement via bug bounty (Wiz). Mise à jour vers GHES 3.19.3 requise.
- **CVE-2026-31431 "Copy Fail" — escalade root Linux** (7.5/10) — Vulnérabilité permettant à un script Python de 732 octets d'obtenir les droits root sur toutes les distributions majeures depuis 2017 (Amazon Linux, RHEL, SUSE, Ubuntu) en manipulant un binaire setuid. CVSS 7.8.
- **CVE-2026-33825 "BlueHammer" — Microsoft Defender** (7.5/10) — Elévation de privilèges SYSTEM pour un attaquant local non-privilégié, patchée lors du Patch Tuesday d'avril 2026 ; déjà ajoutée au catalogue KEV de la CISA.
- **CVE-2026-41940 — cPanel authentication bypass** (7/10) — Contournement d'authentification non-authentifié CVSS 9.8 sur cPanel/WHM v. > 11.40 ; accès direct au panneau de contrôle d'hébergement.
- **CISA KEV — SimpleHelp, Samsung MagicINFO 9, D-Link DIR-823X** (7/10) — Quatre vulnérabilités exploitées activement ajoutées au catalogue KEV avec deadline fédérale mai 2026.
- **Ransomware + zero-days : +143 % de victimes** (7/10) — Dark Reading : les groupes ransomware exploitent de plus en plus les zero-days et one-day flaws au lieu du phishing classique ; Storm-2460 a exploité un bug Windows avant le patch d'avril 2025 pour déployer du ransomware.
- **CVE-2025-14733 — WatchGuard Firebox RCE** (6.5/10) — Out-of-bounds write dans Fireware OS permettant l'exécution de code à distance ; threat actors exploitent activement la faille selon Dark Reading.

## Trends émergents observés

1. **L'écosystème IA/LLM devient une cible supply chain de premier plan** — Deux incidents majeurs cette semaine visent directement l'outillage IA : PyTorch Lightning compromis sur PyPI, LiteLLM exploité en production. Les clés d'API LLM (OpenAI, Anthropic) sont désormais des actifs aussi critiques que les credentials cloud. Les équipes sécurité doivent intégrer les registres de packages IA dans leur périmètre de surveillance, au même titre que npm ou Maven.

2. **Fenêtre d'exploitation post-divulgation en contraction rapide** — CVE-2026-42208 (LiteLLM) exploitée en 36 heures, Lightning (PyPI) détecté en 18 minutes par un scanner automatique. Les processus de patch management traditionnels (cycle 30/60/90 jours) sont structurellement inadaptés à cette temporalité. La réponse efficace passe par l'automatisation du déploiement des patchs critiques et l'instrumentation des détections comportementales.

3. **Harmonisation réglementaire NIS2/DORA/RGPD en phase d'accélération** — L'avis conjoint CEPD 4/2026, le ReCyF ANSSI, et l'horizon juillet 2026 pour la loi Résilience convergent vers un régime de conformité cyber unifié en France. Les entités multi-sectorielles (ex : groupe industriel avec filiale financière) vont devoir aligner leurs procédures de notification sur un guichet unique, ce qui constitue un chantier governance majeur pour 2026.

4. **Identité comme surface d'attaque principale dans les pipelines DevSecOps** — La compromission Lightning démontre que les tokens GitHub sont aujourd'hui la clé maîtresse des pipelines CI/CD : avec un seul token, les attaquants accèdent à des dizaines de repos et propagent leur payload comme un ver. Le principe de moindre privilège sur les tokens GITHUB_TOKEN et les scoped PAT doit devenir un contrôle non-négociable dans toute politique DevSecOps.

### Draft LinkedIn — CVE-2026-3854 : exécution de code sur GitHub avec un simple git push (score 8/10)

> Un seul `git push`. C'est tout ce qu'il fallait pour exécuter du code arbitraire sur les serveurs de GitHub Enterprise. CVE-2026-3854, CVSS 8.7, découverte par les chercheurs de Wiz et signalée via bug bounty.
>
> Le mécanisme : pendant un `git push`, les valeurs des push options fournies par l'utilisateur n'étaient pas correctement assainies avant d'être insérées dans des headers de services internes. En injectant un délimiteur dans la valeur d'une push option, n'importe quel utilisateur authentifié pouvait faire exécuter des commandes arbitraires côté serveur GitHub.
>
> Ce qui mérite d'être souligné, c'est la réponse de GitHub : moins de 2 heures entre le signalement et le déploiement du fix sur github.com, suivi d'une investigation forensique concluant à zéro exploitation. C'est un modèle de réponse aux incidents que l'on aimerait voir plus souvent, y compris côté investigation post-patch.
>
> En revanche, pour GitHub Enterprise Server, la correction nécessite une mise à jour vers GHES 3.19.3. Ce type de faille illustre pourquoi les entreprises qui hébergent leur propre instance GHES doivent maintenir un processus de mise à jour agile : les CVE critiques sur des outils DevOps ne peuvent pas attendre les cycles de patch classiques.
>
> Pour nous, futurs professionnels cyber, c'est un rappel que la surface d'attaque DevSecOps englobe aussi bien les dépendances applicatives que l'infrastructure git elle-même.
>
